# v8.5 queue — convert silent error swallowing to Sentry reports

Background: the v8.3 → v8.4 cycle exposed that many `catch {}` and
`console.warn` calls swallow errors that should have surfaced. v8.4
wired the root `ErrorBoundary` to Sentry. v8.5 should walk the
identified call sites and convert each `console.warn` to a
`reportError(e, { source: '...' })` so production failures land in
Sentry instead of disappearing.

Total occurrences as of v8.4: **23** across 8 files. Categorised below
into KEEP (intentional fallbacks) and CONVERT (silent failures that
mask bugs).

---

## KEEP — intentional fallbacks (do not touch)

These `console.warn` calls are the Sentry-unavailable fallback path
inside `lib/errorReporting.ts` itself, or in modules that legitimately
treat the operation as best-effort. Touching them would create a loop
or hide non-bugs.

| File | Why keep |
|---|---|
| `lib/errorReporting.ts` (4) | This module IS the Sentry wrapper; its console.warn IS the fallback path when Sentry isn't loaded (Expo Go, missing DSN). |
| `lib/entitlements.ts` configureRevenueCat warn | Documented as no-op when keys aren't set — `__DEV__` gated. |
| `lib/circuitBreaker.ts` (2) | Breaker state transitions; warns are diagnostic, not error reports. |

---

## CONVERT — silent failures that mask bugs

For each: replace the silent catch / console.warn with `reportError(e,
{ source: '<file>:<function>', ...context })`. Where the caller already
handles user-visible feedback (e.g. an Alert), still report so Sentry
sees the trend.

### lib/

- [ ] `lib/entitlements.ts` — `restorePurchases` catch warn → `reportError(e, { source: 'entitlements:restorePurchases' })`. This is a critical IAP path; failures here are the reason "restore" silently does nothing.
- [ ] `lib/entitlements.ts` — `findPackageForPlan` catch warn (line ~273). Already used to silently null-return.
- [ ] `lib/venueSearchApi.ts` (2) — `searchVenues` + `searchAddress` log lines. Convert console.error → `reportError` so the v8.3/v8.4 "venue search broken" reports stop being mystery investigations.

### app/

- [ ] `app/create-watch-party.tsx` (4 occurrences) — location, geocode, manual address, contact-picker error paths. The "Search temporarily unavailable" message hides whether it's coord cascade, Nominatim, Overpass, or something else.
- [ ] `app/watch-party/[id].tsx` (2) — RSVP RPC catch + auto-join catch. These were the v8.3 silent-fail RSVP bug. WatchPartyCard already surfaces via Alert in v8.4; the screen-level handler should follow.

### components/

- [ ] `components/WCFanGroups.tsx` (3) — join group catch + fetch catch. Silent fails mean the user sees the spinner stop with no feedback.
- [ ] `components/WCWatchParties.tsx` (2) — RSVP catch + fetch catch. Same pattern as WatchPartyCard.

## Acceptance criteria for v8.5

- Sentry's "Top Issues" page shows real production errors within 24h
  of a v8.5 install on a real device.
- No regressions in the v8.4 UAT checklist.
- 00b smoke test still passes (`qa/maestro/00b_tab_crash_regression.yaml`).
- This document moved to `qa/v8.5-error-conversion-done.md` with
  checkboxes ticked.

## Out of scope for v8.5

- Replacing `console.warn` calls that fire only inside `__DEV__`
  guards (those don't reach prod logs anyway).
- Sentry breadcrumbs on every navigation tap (nice-to-have, defer
  to v8.6).
- Switching from Sentry to OTel / a different vendor (no signal
  this is needed).
