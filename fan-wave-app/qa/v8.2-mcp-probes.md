### Probe 2 — avatars RLS policies (4 of them) scoped to bucket_id='avatars'

Verifies the public-read + per-user owner write policies are in place. Without these, either anonymous profile-pic loads break (no SELECT) or any signed-in user could overwrite anyone else's avatar (no scoped INSERT/UPDATE/DELETE).

```sql
SELECT policyname, cmd, qual::text AS using_clause, with_check::text AS check_clause
  FROM pg_policies
 WHERE schemaname = 'storage'
   AND tablename = 'objects'
   AND policyname LIKE 'avatars_%'
 ORDER BY policyname;
```

Expected: exactly 4 rows.
- `avatars_public_read` — `cmd='SELECT'`, USING contains `bucket_id = 'avatars'`
- `avatars_owner_insert` — `cmd='INSERT'`, WITH CHECK contains `bucket_id = 'avatars'` AND `storage.foldername(name))[1] = (SELECT auth.uid()::text)`
- `avatars_owner_update` — `cmd='UPDATE'`, USING + WITH CHECK both contain the same predicate as insert
- `avatars_owner_delete` — `cmd='DELETE'`, USING contains the same predicate

Any policy missing OR scoped to a different bucket = 057 did not apply cleanly.

### Probe 5 — `rsvp_to_watch_party` is a single 2-arg overload

Migration 059 dropped the legacy 3-arg signature that no client was calling (every RSVP was 404-ing through PostgREST) and replaced it with a 2-arg overload that reads `auth.uid()` internally + inlines `has_wc_access()` so SECURITY DEFINER can't be used as a bypass route.

```sql
SELECT p.proname,
       pg_get_function_identity_arguments(p.oid) AS args,
       p.prosecdef AS is_security_definer
  FROM pg_proc p
  JOIN pg_namespace n ON n.oid = p.pronamespace
 WHERE n.nspname = 'public'
   AND p.proname = 'rsvp_to_watch_party';
```

Expected: exactly ONE row.
- `args` = `p_party_id uuid, p_status text`
- `is_security_definer` = `true`

If TWO rows come back (the 3-arg overload still exists), migration 059 did not run to completion or a later migration re-created it.

### Probe 6 — `games` table has ≥1 Soccer Cup row for today

End-to-end proof that the full chain works:
059-corrected URL → edge function v4 with the new `worldcup` mapping → ESPN scoreboard → games insert with `league_id` resolved to "Soccer Cup".

> **Prerequisite:** this probe is only meaningful AFTER the first `espn_sync_worldcup_fast` run completes (≤2 min from when cron resumes). If you just applied 058, wait 2-5 minutes before running this probe. If you run this BEFORE the cron fires you'll get 0 rows and false-fail.

```sql
SELECT g.id,
       g.scheduled_at,
       g.status,
       l.name AS league_name,
       ht.name AS home_team,
       at.name AS away_team
  FROM public.games g
  JOIN public.leagues l ON l.id = g.league_id
  LEFT JOIN public.teams ht ON ht.id = g.home_team_id
  LEFT JOIN public.teams at ON at.id = g.away_team_id
 WHERE l.name = 'Soccer Cup'
   AND g.scheduled_at::date = CURRENT_DATE
 ORDER BY g.scheduled_at ASC
 LIMIT 10;
```

Expected: ≥ 1 row, AS LONG AS at least one Soccer Cup match is scheduled for today on espn.com/soccer/fifa.world. Cross-check against the public ESPN schedule before failing this probe — there is no DB-side issue if ESPN itself shows no fixtures today.
